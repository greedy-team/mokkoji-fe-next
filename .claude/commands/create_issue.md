# GitHub 이슈 생성 + 브랜치 연결 워크플로우

- 레포지토리: `greedy-team/mokkoji-fe-next`
- 브랜치 컨벤션: `{이슈번호}-{타입}-{짧은-영문-설명}` (예: `541-feat-club-bookmark-api`)
- **브랜치 베이스**: 항상 `develop`에서 분기한다. (현재 체크아웃된 브랜치나 main에서 분기하지 않는다)

## 컨텍스트 확인

이 커맨드는 두 가지 방식으로 실행된다:

1. **단독 실행** — 사용자가 `/create_issue`를 직접 입력한 경우. 이슈 제목·내용·브랜치 타입을 대화로 수집한다.
2. **plan_from_discord 연계** — 대화 컨텍스트에 이미 플랜이 있는 경우. 해당 플랜 내용을 그대로 활용한다.

`$ARGUMENTS`가 있으면 이슈 제목 초안으로 사용한다.

## 1. 이슈 초안 작성

- 아래 형식으로 이슈 초안을 작성한다:

  **제목**: 작업 내용을 한 줄로 (한국어, 간결하게)

  **본문**:
  ```
  ## 배경
  {작업이 필요한 이유 또는 참조 PR/디스코드 글 요약 1~2줄}

  ## 작업 항목
  - [ ] {항목 1}
  - [ ] {항목 2}
  ...
  ```

- 초안을 사용자에게 보여주고 **멈춘다.** 수정 요청이 오면 반영해 다시 보여준다.

## 2. 이슈 생성

- 승인되면 아래 명령으로 이슈를 생성한다:
  ```bash
  gh issue create --repo greedy-team/mokkoji-fe-next --title "..." --body "..."
  ```
- 출력 URL에서 이슈 번호를 파악한다.

## 3. 브랜치 타입 결정

작업 성격에 따라 타입을 결정한다:

| 성격 | 타입 |
|------|------|
| 신규 기능 구현 | `feat` |
| 버그 수정 | `fix` |
| 설정·정리·의존성 | `chore` |
| 코드 구조 개선 | `refactor` |
| 스타일 조정 | `style` |

## 4. 브랜치 생성

- `{이슈번호}-{타입}-{짧은-영문-설명}` 형식의 브랜치명을 제안한다.
  - 영문 설명은 kebab-case, 3단어 이내로 (예: `club-bookmark-api`)
- 사용자에게 보여주고 확인받는다.
- 승인되면 **반드시 `develop`에서 분기**한다 (최신 상태로 갱신 후 생성):
  ```bash
  git checkout develop
  git pull origin develop
  git checkout -b {브랜치명}
  ```
- 브랜치 생성 및 전환 완료를 알린다.

## 안전 규칙

- `gh issue create`와 `git checkout -b`는 **사용자의 명시적 확인 없이는 절대 실행하지 않는다.**
- 브랜치는 **항상 `develop`에서 분기한다.** 다른 브랜치에서 분기하면 머지되지 않은 커밋이 섞이므로 금지.
- 이미 같은 이름의 브랜치가 존재하면 사용자에게 알리고 다른 이름을 제안한다.
